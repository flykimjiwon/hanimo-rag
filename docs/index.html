Architecture

Two phases: Indexing (batch, offline) and Search (real-time, agentic).

Indexing Pipeline

1

Parse

Document → plain text. Supports PDF, DOCX, Markdown, TXT, and more.

2

Chunk

Recursive text splitting with configurable size and overlap. Same proven strategy.

3

LLM Extract Keys

Small LLM reads each chunk and outputs structured metadata:
topics, entities, questions, category, summary

4

Store

Save to JSON file or SQLite. Human-readable, editable, debuggable.

Agentic Search Loop

Query: "FastAPI middleware CORS error" | v [LLM Router] ← parse query intent + extract search keys | keys: ["fastapi", "middleware", "cors"] v [Key Lookup] ← hash map / SQLite match (O(1) ~ O(log n)) | 15 candidate chunks v [LLM Judge] ← batch relevance check (yes/no per chunk) | 4 relevant chunks | | if not enough ↓ | [LLM Expand] ← suggest: ["starlette", "cors-headers", "allow-origins"] | v [Key Lookup] ← round 2 with expanded keys | v Results (max 3 rounds)

Usage

3 lines to RAG. Same API in Python, Node.js, and Next.js.

Python pip install hanimo-rag 
from hanimo-rag import hanimo-rag

rag = hanimo-rag("qwen2.5:7b")

# Index your documents (batch — run overnight)
await rag.index("./docs")

# Search with agentic loop
results = await rag.search("How to add CORS middleware?")

# Or just ask
answer = await rag.ask("What is the default chunk size?")
print(answer)
TypeScript npm install hanimo-rag 
import { hanimo-rag } from 'hanimo-rag'

const rag = new hanimo-rag({ model: 'qwen2.5:7b' })

// Index
await rag.index('./docs')

// Search
const results = await rag.search('How to add CORS middleware?')

// Ask
const answer = await rag.ask('What is the default chunk size?')
console.log(answer)
Next.js app/api/search/route.ts 
import { hanimo-rag } from 'hanimo-rag'
import { NextResponse } from 'next/server'

const rag = new hanimo-rag({
  model: 'qwen2.5:7b',
  storePath: './data/rag-index'
})

export async function POST(req: Request) {
  const { query } = await req.json()
  const answer = await rag.ask(query)
  return NextResponse.json({ answer })
}

Batch Indexing Strategy

Index during idle hours. Search is instant anytime.

Indexing (Heavy, Batch)

When: Overnight, idle GPU hours
Cost: ~500ms per chunk (7B model)
1,000 chunks: ~8 minutes
10,000 chunks: ~83 minutes

Perfect for cron jobs:
0 3 * * * hanimo-rag index ./docs

Search (Light, Real-time)

When: Any time, on demand
Key Lookup: 1ms (hash map)
LLM Calls: 2x ~150ms (router + judge)
Total: ~350ms per search

No GPU needed for search.
Even a 3B model can route and judge.
