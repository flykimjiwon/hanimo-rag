Part 1. RAG가 뭔가요?

1.1 RAG = "찾아서 답하기"

RAG (Retrieval-Augmented Generation)는 AI가 답변하기 전에 먼저 관련 자료를 찾아보고, 그 자료를 근거로 답하는 방식입니다.

도서관에서 리포트 쓰는 것과 같습니다. 머릿속 기억만으로 쓰면 틀릴 수 있지만, 관련 책을 먼저 찾아서 읽고 쓰면 정확한 리포트가 나옵니다.
사용자 질문: "pgvector 성능은?"
    ↓
  [찾기] → DB에서 관련 문서 조각 3~5개 꺼내옴
    ↓
  [조합] → "이 자료를 참고해서 답변해" + 찾은 자료 → AI에게 전달
    ↓
  [답변] → 근거 있는 답변 생성

왜 필요한가?

  • AI의 학습 데이터가 오래됨 → 최신 정보를 실시간으로 제공
  • AI가 모르면서 아는 척 답하는 것(지어내기) 방지 → 근거 기반 답변
  • 회사 내부 문서, 법률, 의료 같은 전문 지식 주입
  • AI 모델을 다시 학습시키지 않고도 자료만 업데이트하면 됨

1.2 RAG의 발전 단계

요리에 비유하면: 1세대는 "레시피대로만", 2세대는 "재료 손질도 꼼꼼히", 3세대는 "레고처럼 조립식 주방", 4세대는 "AI 셰프가 알아서 판단하며 요리"
세대이름쉽게 말하면한계
1세대 Naive RAG
(기본형)		 문서 쪼개기 → 숫자로 바꾸기 → 비슷한 거 찾기 → AI에게 주기. 그냥 순서대로 하는 것 질문과 답변의 표현이 다르면 못 찾음
2세대 Advanced RAG
(개선형)		 찾기 전에 질문을 다듬고, 찾은 후에 결과를 걸러내는 단계 추가 여전히 순서가 고정됨
3세대 Modular RAG
(조립식)		 레고 블록처럼 각 단계를 떼었다 붙였다 교체 가능. 상황에 따라 다른 조합 사용 설계가 복잡해짐
4세대 Agentic RAG
(자율형)		 AI가 스스로 판단해서 "지금 검색해야 하나?", "결과가 충분한가?", "다시 찾아볼까?" 결정 비용, 속도
hanimo-rag의 위치: 2세대(개선형) + 3세대(조립식) 사이. 의미검색+키워드검색+관계망 3종 검색, 가상 답변 기법, 의미 기반 쪼개기 등을 구현하면서, 부품(Embedder, LLM, Chunker 등)을 자유롭게 교체할 수 있는 구조입니다.

1.3 문서 쪼개기 (Chunking)

긴 문서를 AI가 소화할 수 있는 작은 조각으로 나누는 것. 검색 품질의 기초를 결정합니다.

책 한 권을 통째로 들고 다닐 수 없으니, 필요한 페이지만 뽑아서 가져가는 것과 같습니다. 어떻게 나누냐에 따라 "필요한 내용이 잘리는지" vs "필요 없는 내용이 끼는지"가 달라집니다.
방법쉽게 말하면장점단점hanimo-rag
글자 수 기준 500자마다 자르기. 잘리는 부분은 조금 겹치게 단순, 빠름 문장 중간에 잘릴 수 있음 —
문단/문장 기준
(재귀적)		 먼저 문단으로 나누고, 문단이 너무 길면 문장으로, 문장도 길면 단어로 나눔 문맥이 잘 보존됨 구분자에 의존 적용됨
의미 기준
(시맨틱)		 연속된 문장의 "느낌이 달라지는 지점"에서 자름 주제별로 깔끔하게 나뉨 모든 문장을 숫자로 바꿔야 해서 느림 적용됨
페이지 기준 PDF의 페이지 단위로 자름 원본 구조 유지 페이지 길이가 제각각 적용됨
RAPTOR
(계층형)		 조각들을 묶어서 요약본을 만들고, 요약본끼리 또 묶어서 트리 구조로 만듦 큰 그림도 작은 디테일도 검색 가능 AI 요약 비용 발생 —

1.4 글을 숫자로 바꾸기 (Embedding)

글의 "의미"를 숫자 좌표로 변환하는 것. 이래야 컴퓨터가 "이 글과 저 글이 비슷한지" 계산할 수 있습니다.

지도에 핀 꽂기와 같습니다. "강아지 산책"이라는 글과 "반려견 운동"이라는 글은 단어는 다르지만 의미가 비슷하니까, 지도에서 가까운 곳에 핀이 꽂힙니다. "주식 투자"는 의미가 다르니 멀리 떨어져 꽂힙니다.

의미 기반 (Dense)

AI 모델이 글 전체의 "느낌"을 하나의 좌표로 압축

잘하는 것: "비슷한 뜻의 다른 표현" 찾기

못하는 것: 정확한 단어 매칭 (제품명, 코드 등)

hanimo-rag: Ollama + OpenAI 지원

키워드 기반 (Sparse)

어떤 단어가 몇 번 나오는지를 기반으로 점수 계산 (BM25 방식)

잘하는 것: 정확한 단어, 제품명, 코드 검색

못하는 것: 같은 뜻 다른 단어 ("차량" vs "자동차")

hanimo-rag: PostgreSQL 내장 검색

핵심: 의미 기반 + 키워드 기반을 합치면 단독보다 검색 정확도가 18~22% 향상됩니다. hanimo-rag는 여기에 관계망 검색까지 3종 결합합니다.

1.5 검색하는 3가지 방법

의미 검색 (Vector)

글의 "느낌"이 비슷한 문서를 찾음

"이것과 비슷한 느낌의 글 찾아줘"

예: "강아지 건강" 검색 → "반려견 질병 예방" 찾아줌

키워드 검색 (FTS)

정확한 단어가 포함된 문서를 찾음

Ctrl+F처럼 정확한 단어 찾기

예: "pgvector" 검색 → 정확히 "pgvector"가 적힌 문서

관계망 검색 (Graph)

사람/개념/사물 간의 연결 관계를 따라가며 찾음

"친구의 친구"를 찾는 것. A→B→C 연결을 따라감

예: "이 프로젝트의 담당자의 부서는?" → 관계 추적

1.6 검색 결과 합치기 (RRF - 순위 합산)

의미 검색, 키워드 검색, 관계망 검색 — 3가지 방법으로 각각 검색하면 결과 3개가 나옵니다. 이걸 하나의 최종 순위로 합치는 방법이 RRF입니다.

3명의 심사위원이 각자 후보를 순위 매긴 뒤, 세 명의 순위를 합산해서 최종 당선자를 뽑는 것과 같습니다. 한 심사위원이 100점 만점, 다른 심사위원이 10점 만점이어도 상관없이 "몇 등인가"만 봅니다.
점수 계산:  각 문서의 점수 = 합계( 1 / (60 + 순위) )

예시: 문서 A가 의미검색 1등, 키워드검색 3등, 관계망 검색 2등이면
 = 1/(60+1) + 1/(60+3) + 1/(60+2)
 = 0.0164 + 0.0159 + 0.0161 = 0.0484  ← 높은 점수!

장점: 각 검색 방법의 점수 단위가 달라도 상관없이 공정하게 합칠 수 있음

hanimo-rag 적용 3개 검색 결과를 가중치 있는 순위 합산으로 결합

1.7 가상 답변 기법 (HyDE)

질문을 그대로 검색하는 대신, AI에게 "이 질문의 답이 뭘까?" 가짜 답을 먼저 만들게 하고, 그 가짜 답과 비슷한 진짜 문서를 찾는 기법.

수배범을 찾을 때 "키 180cm 남성"이라는 설명 대신, 몽타주(가상 얼굴)를 먼저 그려서 그 얼굴과 비슷한 사람을 찾는 것과 같습니다.
질문: "pgvector 성능은?"
    ↓ AI가 가상 답변 생성
가짜 답: "pgvector는 HNSW 알고리즘으로 초당 수만 건 검색이 가능하며..."
    ↓ 이 가짜 답을 숫자로 변환
가짜 답의 좌표로 진짜 문서 검색 → 더 정확한 결과!

왜 효과적? 짧은 질문보다 긴 가짜 답이 진짜 답변 문서와 "느낌"이 더 비슷하기 때문

hanimo-rag 적용 core/hyde.py

1.8 관계망 검색 (Graph RAG)

문서에서 사람, 개념, 제품 같은 "것들"(엔티티)과 그것들 사이의 "관계"를 뽑아내서 인물관계도 같은 그래프를 만들고, 이 관계를 따라가며 검색하는 방식.

위키백과의 하이퍼링크와 같습니다. "서울" 문서에서 "한국" 링크를 클릭하고, "한국"에서 "대통령" 링크를 클릭하면 — 이렇게 연결을 따라가면 직접 검색으로는 못 찾을 정보도 찾을 수 있습니다.

처리 과정

문서 조각 → AI가 읽고 "것들"과 "관계" 추출 → 관계도 저장
                                                    ↓
질문 → 질문과 가장 관련 있는 "것" 찾기 → 그 "것"의 친구/친구의 친구 탐색 → 결과

언제 쓰면 좋나?

상황의미 검색만관계망 검색
"매출 보고서 보여줘" (단순 조회)충분과잉
"이 결재를 승인한 사람의 부서장은?" (연결 추적)어려움적합
"전체 문서에서 주요 주제 3개는?" (전체 요약)어려움적합

hanimo-rag 적용 AI 기반 관계 추출 + "친구의 친구" 2단계 탐색 + [[위키링크]] 자동 인식

1.9 검색 결과 재정렬 (Reranking)

1차 검색으로 후보 50~100개를 빠르게 골라낸 뒤, 더 정밀한 AI 모델로 상위 10개를 다시 순위 매기는 것.

서류 심사(1차)로 50명 합격시킨 뒤, 면접(2차)으로 최종 10명을 뽑는 것과 같습니다. 서류 심사는 빠르지만 대략적이고, 면접은 느리지만 정확합니다.
[추천 흐름]
키워드검색 + 의미검색 → 순위 합산(RRF) → 재정렬(상위 10개 정밀 평가) → AI에게 전달

1.10 질문 바꿔서 다시 찾기 (Query Transformation)

방법쉽게 말하면예시
여러 버전으로 질문
(Multi-Query)		 같은 질문을 다양한 표현으로 바꿔서 각각 검색. 결과 합침 "RAG 성능" → "RAG 정확도", "RAG 속도", "RAG 벤치마크" 3가지로 검색
질문 쪼개기
(Decomposition)		 복잡한 질문을 쉬운 소질문 여러 개로 나눔 "A와 B 비교" → "A는?", "B는?" 각각 찾고 합침
한 발짝 물러서기
(Step-Back)		 너무 구체적인 질문을 한 단계 넓은 질문으로 바꿈 "아이폰 주사율" → "아이폰 기술 사양" (더 넓은 문서를 찾을 수 있음)
가상 답변 (HyDE) 질문을 가짜 답변으로 바꿔서 검색 (위 1.7 참조) 짧은 질문 → 긴 가짜 답변 → 진짜 답변과 비슷한 느낌으로 검색

1.11 최신 연구 논문들

RAPTOR (2024, 스탠포드)

문서 조각들을 묶어서 요약 → 요약을 또 묶어서 요약하여 트리(나무) 구조를 만듦. 세부 내용도, 전체 요약도 모두 검색 가능.

회사 보고서의 "요약본 → 팀별 보고서 → 개인 보고서" 계층 구조와 같습니다.

성과: 정확도 20% 향상

CRAG - 교정형 RAG (2024)

검색 결과가 좋은지 나쁜지 스스로 판단하고, 나쁘면 웹 검색으로 보완.

오픈북 시험에서 찾은 자료가 답과 안 맞으면, 다른 책을 찾아보는 것.

좋으면 그대로 사용 / 나쁘면 웹 검색 / 애매하면 둘 다 합침

Self-RAG (2023)

AI가 "지금 검색이 필요한가?"를 스스로 판단. 쉬운 질문은 바로 답하고, 어려운 질문만 검색.

선생님이 학생 질문에 "이건 내가 아니까 바로 답" vs "이건 교과서 확인하고 답할게" 구분하는 것.

Adaptive RAG (2024)

질문 난이도에 따라 검색 전략을 다르게 적용. 쉬우면 검색 안 함, 보통이면 1번 검색, 어려우면 여러 번 반복 검색.

네비게이션이 "길이 단순하면 안내 안 하고, 복잡한 교차로만 안내하는 것"과 같습니다.

Part 2. hanimo-rag는 어떤 프로젝트인가?

2.1 핵심 철학

"PostgreSQL 하나로 전부 한다"

보통 RAG를 만들려면 여러 가지 서비스를 따로 깔아야 합니다. hanimo-rag는 PostgreSQL 하나에 전부 넣었습니다.

다른 RAG: 냉장고 따로, 오븐 따로, 전자레인지 따로 사서 주방 구성.
hanimo-rag: 냉장고+오븐+전자레인지 일체형 하나만 사면 끝.
보통 RAG 스택 (4개 서비스): 
PostgreSQL (기본 데이터)
+ Pinecone (의미 검색용)
+ Elasticsearch (키워드 검색용)
+ Neo4j (관계 검색용)
= 4개를 각각 설치/관리
hanimo-rag (1개 서비스): 
PostgreSQL 하나에 전부 포함
  - pgvector 확장 (의미 검색)
  - 내장 tsvector (키워드 검색)
  - 내장 SQL (관계 검색)
= 1개만 관리하면 끝
핵심 가치쉽게 말하면
관리 단순PostgreSQL 하나만 운영. 백업도 모니터링도 1개만
데이터 안전의미검색 데이터와 관계망 데이터가 같은 DB → 동시에 저장/취소 가능 (돈 이체처럼 "보내기+받기"가 동시에 되거나, 둘 다 안 되거나)
설치 간편pip install hanimo-rag 한 줄이면 끝
어디든 붙임Supabase, AWS RDS 등 표준 PostgreSQL이면 다 됨
의존성 없음LangChain, LlamaIndex 같은 무거운 프레임워크 없이 직접 구현
2가지 사용법독립 서버(REST API)로도, 라이브러리(import)로도 사용 가능

2.2 왜 PostgreSQL 하나만?

기능PostgreSQL에서 어떻게?전용 제품 대비
의미 검색 pgvector 확장 + 빠른 검색 목록(HNSW) + 절반 크기 저장(halfvec) Pinecone/Weaviate와 비슷한 정확도. 수천만 건까지 실용적
키워드 검색 PostgreSQL 내장 기능 (tsvector). 별도 설치 불필요 Elasticsearch 대비 관리 비용 0. 중소 규모에 충분
관계 탐색 SQL로 "친구의 친구" 연쇄 탐색 (Recursive CTE) Neo4j보다 기능은 적지만, 2단계 탐색에는 충분
데이터 안전 PostgreSQL 기본 기능 여러 DB 쓰면 "A는 저장됐는데 B는 안 됐다" 문제 발생

2.4 처리 흐름

문서 넣기 (Ingest)

도서관에 새 책이 들어오면: 읽기 → 요약 카드 만들기 → 분류 번호 붙이기 → 서가에 꽂기 → 관련 책 연결
파일 업로드
  → 읽기 (PDF/Word/... → 텍스트 추출)
    → 쪼개기 (512자 단위, 문단/의미/페이지 기준)
      → 숫자로 바꾸기 (Ollama 또는 OpenAI, 32개씩 묶어서 처리)
        → 저장 (PostgreSQL에 벡터 + 키워드 인덱스 자동 생성)
          → 관계 추출 (AI가 사람/개념/관계를 뽑아냄)
            → 관계도 저장 (노드/연결선 저장)

검색하기 (Search)

3명의 사서에게 동시에 질문하고, 세 명의 답을 합쳐서 최종 답을 만드는 것
질문 입력
  → 질문을 숫자로 변환
  → (선택) 가상 답변 기법으로 질문 보강
  → 사서 1: 의미 검색 (느낌이 비슷한 것 찾기)
  → 사서 2: 키워드 검색 (정확한 단어 찾기)
  → 사서 3: 관계망 검색 (연결 따라가기)
  → 세 결과의 순위 합산 (RRF)
  → 최종 상위 결과 반환

답변 만들기 (Generate)

검색 결과 (상위 조각들)
  → "다음 자료를 참고해서 답해주세요" + 자료 + 질문 → AI에게 전달
  → 글자가 하나씩 실시간으로 나옴 (ChatGPT처럼)

Part 3. 코드에서 발견된 문제와 수정 내역

심각도문제쉽게 말하면수정 여부
치명적 관계망 노드에 좌표(임베딩) 누락 관계도의 각 인물에 "위치 정보"가 없어서 관계망 검색이 아예 안 됨 수정 완료
치명적 JSON 변환 버그 데이터를 저장할 때 형식이 깨져서 DB에서 읽을 때 에러 발생 가능 수정 완료
중요 순위 합산에 가중치 미적용 의미검색/키워드검색/관계검색의 비중을 조절할 수 없었음 수정 완료
중요 관계 거리에 따른 감쇠 미적용 "친구"와 "친구의 친구"가 같은 중요도로 취급됨 수정 완료
개선 숫자변환 비효율 Ollama 호출할 때 매번 새 연결을 만듦 → 느림 향후
개선 로그 없음 에러가 나도 어디서 났는지 기록이 안 됨 향후
개선 설정 캐싱 없음 설정 파일을 매번 다시 읽음 (한 번만 읽으면 됨) 향후
개선 임시 파일 미삭제 문서 업로드 후 임시 파일이 쌓임 향후
개선 키워드 검색 영어만 키워드 검색이 영어 기준으로 고정되어 한국어 검색이 약함 향후
